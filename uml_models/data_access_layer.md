# Data Access Layer

The Data Access Layer is the boundary between the external signal generator and the internal CHMS subsystems. Its purpose is to receive raw data from supported transports, convert that data into a standard internal representation, and hand it over without exposing downstream components to transport-specific details. This is essential in a cardiovascular monitoring system because simulator inputs may change over time, but the rest of CHMS should continue to operate on stable internal objects.

`DataListener` defines the common contract for receiving raw data. `TCPDataListener`, `WebSocketDataListener`, and `FileDataListener` are separate implementations because each transport has different connection behavior, but all expose the same operations to the adapter. This keeps acquisition logic modular and makes new listener types, such as message queue consumers, easy to add later without affecting existing components.

Parsing is deliberately separated from listening. `DataParser` defines the parsing contract, while `JsonDataParser` and `CsvDataParser` provide format-specific implementations. This prevents transport classes from being overloaded with syntax interpretation logic. A TCP stream and a file can therefore reuse the same parser if they carry the same message format. Both parsers produce the same `ParsedMeasurement` object, which gives the rest of CHMS a consistent data structure.

`DataSourceAdapter` coordinates the listener and parser, validates the resulting measurement, and forwards it through `CHMSInputPort`, an internal handoff interface. This isolates the rest of the system from both transport and format concerns. The result is a modular and maintainable design in which listeners manage connectivity, parsers handle structure, and the adapter controls the boundary to the internal system, supporting future extensibility with minimal coupling.
