# Patient Identification System

The patient identification model is intentionally conservative because identity mistakes in a cardiovascular monitoring context can lead directly to unsafe clinical decisions. `PatientIdentifier` acts as the matching service that evaluates an incoming identifier and demographic context before the rest of the CHMS is allowed to attach measurements or alerts to a patient. It does not own patient records itself; instead, it uses `IdentityManager` as the controlled index of active `HospitalPatient` entries.

The diagram separates patient facts from matching logic. `HospitalPatient` represents the admitted patient record, while `DemographicProfile` captures the set of demographic fields used for comparison. This allows matching rules to evolve without changing how the patient entity is stored. `IdentifierToken` models operational identifiers such as wristband codes or externally issued monitoring IDs, which may need expiry checking or issuer validation before they can be trusted.

`MatchResult` is included to make uncertainty explicit. Rather than forcing every lookup into a binary success or failure outcome, the design allows confidence scoring and review thresholds. This is important when partial identifiers or inconsistent data arrive from external systems. If a match is weak, ambiguous, or contradictory, `PatientIdentifier` raises an `IdentityAnomaly`, and `IdentityManager` records it for review. That provides a formal path for mismatch handling instead of silently attaching data to the wrong patient.

Access rules should remain narrow. Other subsystems should request only the minimum identity confirmation needed for their task, not unrestricted demographic data. Alert generation, for example, should verify that a patient is currently admitted and matched, while richer patient details remain protected within the identity subsystem. This preserves privacy, supports auditing, and keeps identity governance separate from storage and monitoring behavior.
